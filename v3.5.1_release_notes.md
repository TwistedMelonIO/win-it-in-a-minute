# v3.5.1 Release Notes — License Persistence Auto-Test + Bad-Paste Recovery

**Released: April 29, 2026**

## What's New

### License Persistence Auto-Test
- **Summary**: New script `scripts/test-license-persistence.sh` runs 12 automated checks against the dual-tier persistence introduced in v3.5.0.
- **Details**: Covers container restart, rebuild, force-recreate without env, volume destruction (`down -v`), host-backup destruction, and bad-paste rejection. Reports pass/fail with colored output, exits non-zero on any failure. Runs in ~30s.
- **Usage**: `bash scripts/test-license-persistence.sh` — run after every deploy to verify the demo unit will not lose its license.
- **Impact**: Rollout to a new machine now has a one-command verification step.

## Bug Fixes

### Bad-paste no longer poisons in-memory license state
- **Summary**: Pasting an invalid license key in the web UI no longer leaves the running app in an "invalid" state until restart.
- **Details**: The activate endpoint now silently restores the previously-persisted key to `process.env.LICENSE_KEY` whenever the new paste fails validation, then re-validates so `licenseState` returns to valid. The HTTP response still reflects the paste-attempt result so the user sees their bad paste was rejected.
- **Caught by**: The new auto-test (T7 post-revalidate check). Caught the bug on the first run.

## Technical Changes
- `server.js`: activate endpoint now captures paste result before optional silent recovery.
